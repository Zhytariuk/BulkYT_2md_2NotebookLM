# 🤖 AI Context: Поточний стан проекту для наступного ШІ

## 🔩 Технічне середовище
- **OS**: Windows (PowerShell)
- **Python**: 3.8+
- **Бібліотеки**: `yt-dlp`, `requests`
- **Системні залежності**:
    *   `deno` (встановлено через winget) — для JS-челенджів.
    *   `ffmpeg` (встановлено через winget) — для стабільності `yt-dlp`.
- **Файли даних**:
    *   `youtube_cookies.txt` (обов'язковий для обходу 429).
    *   `links_to_process.txt` (список URL).

## 📊 Поточний стан логіки
На даний момент проект розділено на два основні сценарії:
1.  **`yt_links_to_md_transcript_with_dates.py`** — автоматичне витягування (План D/F).
2.  **`yt_links_to_md_titles_with_dates.py`** — створення шаблонів для ручної вставки (швидко та надійно).

### ⚙️ Технічна база (План D/F):
Ми перейшли від `youtube_transcript_api` до **Прямого завантаження** через `yt-dlp` + `requests`:
1.  `yt-dlp` отримує метадані (назву, канал) та прямий URL на субтитри.
2.  Скрипт витягує прямий URL на `json3` (JSON подій) або `vtt` субтитри.
3.  `requests` завантажує файл із куками та браузерним User-Agent.
4.  Парсер очищує отриманий текст від таймкодів та службових тегів.

## ⚠️ Відомі проблеми (Битва за 429)
1.  **n-challenge**: YouTube блокує завантаження EJS-рішень (`skipped`), що заважає `yt_dlp` діставати субтитри самостійно.
2.  **PO-Token**: YouTube вимагає `po_token` для більшості веб-клієнтів.
3.  **Гіпотеза**: Тимчасове блокування за IP-адресою на рівні API субтитрів. Потрібно зачекати 1-2 години.

## 🛠️ Напрямки для наступних кроків
- Якщо 429 не зникне після паузи — спробувати **`curl_cffi`** (утиліта, що імітує TLS-відбиток браузера).
- Реалізувати підтримку **проксі** через `ydl_opts` та `requests` сесію.
- Додати динамічне отримання `PO-Token` (дуже складно, але можливо через браузерні розширення).
